# Story Analyzer — 物語構造分析・開発ツール

Save the Cat 15ビート、6つの感情アーク（Reagan et al. 2016）、三幕構成、起承転結の4フレームワークを統合し、既存ストーリーの診断と新規ストーリーの構造設計を行う。

## 対応フォーマット

| フォーマット | 総尺目安 | 主要構造 |
|---|---|---|
| 映画 | 90-120分 | STC15ビート完全版 |
| ドラマシリーズ | 1話40-60分 × 10-30話 | マクロアーク + エピソード8ビート |
| ショートドラマ | 1話1-3分 × 50-100話 | 超圧縮4ビート + マクロアーク |
| マンガ | 1話16-24頁 | 起承転結 + 巻単位STC |

## 参照ファイル

| ファイル | 内容 |
|---|---|
| `references/frameworks.md` | 4フレームワーク統合定義・相互対応表 |
| `references/emotional-arcs.md` | 6アーク型の詳細（曲線パターン、ジャンル親和性、代表作品） |
| `references/format-guides.md` | フォーマット別ビートタイミング・重み付け |
| `references/diagnosis-rubric.md` | 4カテゴリ×5段階のスコアリング基準 |

## Mode 1: 分析モード（Analysis）

**トリガー**: ファイル（Word/PDF/テキスト/Markdown）を渡して「分析して」「診断して」「構造チェックして」

### 推奨入力フォーマット

| 優先度 | 形式 | 理由 |
|---|---|---|
| 1 | テキスト (.txt) / Markdown (.md) | テキストデータそのもの。読み取り精度100% |
| 2 | Word (.docx) | 縦書きでもテキスト抽出されるため正確 |
| 3 | PDF（テキスト埋め込み） | コピペ可能なPDFなら縦書きでもOK |
| 4 | PDF（画像ベース/スキャン） | 画像認識に依存。縦書き日本語は精度が落ちる |

### 画像ベースPDFを受け取った場合のフロー

画像ベースPDF（テキスト選択不可、特に縦書き日本語）を受け取った場合は、**分析に入る前に**以下を確認する：

1. 「このPDFは画像ベースのため、読み取り精度が落ちる可能性があります」と伝える
2. 「元のWordファイル（.docx）はありますか？ 脚本は通常Wordで書かれているので、あればそちらの方が正確に分析できます」と聞く
3. テキスト埋め込みPDFの有無も確認する
4. どれもない場合のみ、画像ベースPDFで分析を進める。その際、「読み取り精度に限界がある旨」をレポートの冒頭に明記する

### Step 1: 読み取り & パース

- ユーザーが渡したファイルをClaude Codeで直接読み取る
- フォーマットを判定（映画/ドラマ/ショート/マンガ）。不明ならユーザーに確認
- シーン/章/エピソード単位に分割
- 各シーンの要約と感情的トーン（-1.0〜+1.0のバレンス値）を判定

### 絶対ルール（分析の正確性）

1. **脚本に書かれていない展開を推測で補完してはならない。** 存在しないシーンを創作することは分析ではなく捏造であり、絶対に禁止
2. **各ビートのlabelには、脚本の具体的なシーン番号（S1, S2...）を必ず紐づけること。** 根拠のないビート判定は行わない
3. **曖昧な箇所・読み取れなかった箇所は正直に申告する。** 「S○○のシーンはこういう理解で合っていますか？」とユーザーに確認する
4. **読み取り精度が低いと判断した場合（画像ベースPDF等）は、その旨を冒頭で伝える。** 全シーンが正確に読めたと偽らない

### Step 2: フレームワークマッピング

`references/frameworks.md` と `references/format-guides.md` を参照し、4つのフレームワークで同時にマッピング:

1. **Save the Cat ビート**: フォーマットに応じたビートセット（映画=15、ドラマ=8、ショート=4、マンガ=起承転結）の各ビートがどのシーンに対応するか特定。各ビートのstatus判定:
   - `present`: 明確に存在し、適切な位置にある
   - `weak`: 存在するが弱い（タイミングがズレている、インパクト不足）
   - `missing`: 欠落している

2. **感情アーク**: `references/emotional-arcs.md` の6型それぞれとのフィッティングを計算。最も近い型を判定し、理想曲線との主要なズレを特定

3. **三幕構成**: 各幕の比率を計算。標準（25/50/25）との差異を検出

4. **起承転結**: 転（ターニングポイント）のインパクト、結の収束度を評価

### Step 3: スコアリング & 診断

`references/diagnosis-rubric.md` を参照し、4カテゴリで評価:

| カテゴリ | 評価内容 |
|---|---|
| **構造**（Structure） | ビートの充足率と配置の適正さ |
| **アーク明度**（Arc Clarity） | 6型のどれかに明確に乗っているか |
| **ペーシング**（Pacing） | 各ビートのタイミングはフォーマット基準に対して適正か |
| **引き**（Engagement） | フォーマット固有のフック（冒頭、クリフハンガー等） |

各カテゴリ A/B/C/D/F の5段階。`references/format-guides.md` の重み付け表で加重平均し総合評価を算出。C以下のカテゴリには `references/diagnosis-rubric.md` の改善提案テンプレートから具体的な改善案を出力。

### Step 4: 可視化JSON作成 → ストーリーマップ生成

分析結果から以下の構造のJSONを作成し、`/tmp/story_analysis.json` に保存:

```json
{
  "title": "作品タイトル",
  "format": "movie",
  "arc_type": "cinderella",
  "arc_type_ja": "感動（シンデレラ）",
  "arc_ideal": [[0, 0.0], [10, 0.3], [25, 0.6], [50, -0.3], [75, -0.5], [90, 0.7], [100, 0.8]],
  "acts": {"act1_end": 25, "act2a_end": 50, "act2b_end": 75},
  "kishotenketsu": {"ki": "起", "sho": "承", "ten": "転", "ketsu": "結"},
  "beats": [
    {"name": "Opening Image", "name_ja": "オープニング", "position": 1, "valence": -0.2, "status": "present", "label": "シーンの要約"}
  ],
  "scores": {
    "structure": {"grade": "B", "value": 78},
    "arc_clarity": {"grade": "A", "value": 92},
    "pacing": {"grade": "B", "value": 74},
    "engagement": {"grade": "C", "value": 58}
  }
}
```

arc_idealの値は `references/emotional-arcs.md` から判定したアーク型の曲線パターンを使用する。

ストーリーマップ画像を生成:

```bash
python3 ~/.claude/skills/story-analyzer/scripts/visualize_story.py \
  --config /tmp/story_analysis.json \
  --output ~/Desktop/story_map_[タイトル].png
```

## Mode 2: 開発モード（Development）

**トリガー**: 「プロットを作りたい」「ビートシートを作って」「○○型の映画を構成したい」「Cinderella型の復讐ドラマ」

### Step 1: ヒアリング

以下の情報をユーザーから取得する。「★」は必須、「☆」は任意。

- ★ フォーマット: 映画 / ドラマ / ショートドラマ / マンガ
- ★ ジャンル: アクション / ラブコメ / サスペンス / ホラー / 復讐劇 / ヒューマンドラマ etc.
- ☆ 感情アーク型: 6型から選択（未指定の場合は `references/emotional-arcs.md` のジャンル親和性から推薦）
- ☆ 話数（ドラマ/ショートの場合。未指定なら標準値を提案）
- ☆ ログライン（あれば）
- ☆ 主要キャラクター（あれば）
- ☆ テーマ（あれば）

### Step 2: ビートシート生成

`references/format-guides.md` からフォーマット別テンプレートを選択し、感情アーク型に合わせてビートシートを生成。

**映画の場合** — STC15ビート完全版:
```
【Opening Image (0-1%)】感情: -0.2
  役割: 主人公の「ビフォー」を見せる
  → [具体的なシーン提案]

【Theme Stated (5%)】感情: -0.1
  役割: テーマを誰かのセリフで暗示
  → [具体的なシーン提案]
...
```

**ドラマの場合** — 2層構造で生成:
1. シリーズ全体のマクロビートシート（何話目にどのSTC大ビートが来るか）
2. 各話のミニビートシート（8ビート版）

**ショートドラマの場合** — 3層構造:
1. マクロアーク（全体の感情曲線）
2. 5-10話単位のサブアーク
3. 各話の超圧縮4ビート（Hook→Escalation→Turn→Cliffhanger）

**マンガの場合** — 2層構造:
1. 巻単位のSTC構成
2. 各話の起承転結（ページ配分付き）

### Step 3: プロット展開支援（対話的）

ビートシートをベースに、ユーザーと対話しながら各ビートの具体的なシーンを詰めていく。

対話のポイント:
- 各ビートについて「このビートでは○○が起こる必要があります。具体的にどんなシーンをイメージしていますか？」と聞く
- ユーザーのアイデアに対して、感情アーク上の位置づけとフレームワーク的な妥当性をフィードバック
- 「このシーンのバレンスは+0.3ですが、理想曲線では-0.2が期待されています。もう少し暗い展開にするか、このビートの後に落差を入れると効果的です」等の具体的アドバイス

### Step 4: 可視化

開発中のプロットをJSONに変換し、ストーリーマップを随時生成。ビートが埋まるたびに更新可能。

```bash
python3 ~/.claude/skills/story-analyzer/scripts/visualize_story.py \
  --config /tmp/story_dev.json \
  --output ~/Desktop/story_map_dev.png
```

### Step 5: エクスポート

完成したビートシート+プロットをMarkdownで出力:

```bash
# 出力ファイル
~/Desktop/[タイトル]_beatsheet.md    # ビートシート
~/Desktop/[タイトル]_story_map.png   # ストーリーマップ
```

このMarkdownは `entertainment-scripts` スキルでの脚本執筆にそのまま渡せる形式とする。

## 注意事項

- このツールは構造分析・設計の支援ツールであり、脚本の執筆そのものは行わない
- 分析結果は絶対的な評価ではなく、フレームワークに基づく指標。最終的な創作判断はユーザーが行う
- 「ルールを破る」ことが効果的な場合もある。低スコアが必ずしも悪いストーリーを意味するわけではない
- 複数のアーク型が混合している場合や、意図的にフレームワークから逸脱している場合は、その旨をユーザーに確認する
